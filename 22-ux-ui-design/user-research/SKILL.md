# User Research

> **Current Level:** Intermediate  
> **Domain:** UX/UI Design / Product

---

## Overview

User research helps understand user needs, behaviors, and motivations to create better products. Effective user research uses various methods like interviews, surveys, usability testing, and analytics to gather insights that inform product decisions.

## 2. Research Methods

### Qualitative Methods

```markdown
# Qualitative Research Methods

## 1. User Interviews

### Purpose
- Understand user needs
- Explore motivations
- Discover pain points
- Gather insights

### When to Use
- Early in design process
- Exploring new concepts
- Understanding user behavior
- Validating assumptions

### Best Practices
- Prepare questions in advance
- Use open-ended questions
- Listen more than talk
- Record and take notes
- Follow up on interesting points

### Interview Structure
1. Introduction
2. Background questions
3. Exploration questions
4. Deep dive
5. Wrap-up

### Example Questions
- "Tell me about how you currently [task]?"
- "What challenges do you face when [task]?"
- "What would make [task] easier for you?"
- "How do you feel about [current solution]?"

## 2. Usability Testing

### Purpose
- Evaluate usability
- Identify issues
- Measure performance
- Gather feedback

### When to Use
- Testing prototypes
- Evaluating existing products
- Validating designs
- Measuring usability

### Best Practices
- Use realistic tasks
- Recruit representative users
- Think aloud protocol
- Record sessions
- Debrief after test

### Test Structure
1. Introduction
2. Task instructions
3. Task completion
4. Debrief

### Metrics to Measure
- Task completion rate
- Time on task
- Error rate
- Satisfaction score

## 3. Focus Groups

### Purpose
- Gather group insights
- Explore opinions
- Identify common themes
- Generate ideas

### When to Use
- Exploring concepts
- Gathering opinions
- Brainstorming ideas
- Understanding group dynamics

### Best Practices
- Keep group small (6-8)
- Use skilled moderator
- Prepare discussion guide
- Record sessions
- Analyze themes

### Discussion Structure
1. Introduction
2. Topic exploration
3. Activity/discussion
4. Wrap-up

## 4. Contextual Inquiry

### Purpose
- Observe users in context
- Understand environment
- Identify real-world issues
- Gather authentic insights

### When to Use
- Understanding usage context
- Identifying environmental factors
- Exploring workflows
- Validating in-situ

### Best Practices
- Observe without interfering
- Ask questions at appropriate times
- Take detailed notes
- Record observations
- Respect user privacy
```

```markdown
# Quantitative Research Methods

- Gather large-scale data
- Measure user preferences

### When to Use

- Measuring satisfaction
- Identifying patterns

4. Demographics
5. Thank you

### Question Types

- Likert scale

### Purpose
- Compare alternatives
- Measure performance
- Optimize conversions
- Make data-driven decisions

### When to Use
- Testing design variations
- Optimizing features
- Improving conversions
- Validating hypotheses

### Best Practices
- Test one variable at a time
- Use sufficient sample size
- Run tests long enough
- Statistical significance
- Document results

### Metrics to Measure
- Conversion rate
- Click-through rate
- Time on page
- Bounce rate
- Revenue

## 3. Analytics Analysis

### Purpose
- Understand user behavior
- Identify patterns
- Measure performance
- Track goals

### When to Use
- Understanding usage patterns

### Best Practices
- Define goals and KPIs
- Set up proper tracking
- Regular analysis
- Segment data
- Act on insights

### Metrics to Track

- Unique visitors

- Bounce rate
- Conversion rate
- Funnel completion

### Purpose
- Visualize user behavior

- Understanding user attention
```

## 5. Pain Point Identification

### Pain Point Analysis

```markdown
# Pain Point Identification

## Types of Pain Points

### 1. Financial Pain Points
- Product/service too expensive
- Hidden costs
- Poor value for money
- Unexpected charges

### 2. Productivity Pain Points
- Wastes time
- Inefficient processes
- Too many steps
- Complex workflows

### 3. Process Pain Points
- Difficult to use
- Confusing interface
- Poor navigation
- Lack of features

### 4. Support Pain Points
- Poor customer service
- Lack of documentation
- Slow response times
- Unhelpful support

## Pain Point Discovery Methods

### 1. User Interviews
- Ask about challenges
- Explore frustrations
- Identify workarounds
- Understand context

### 2. Usability Testing
- Observe struggles
- Note errors
- Measure time
- Track completion

### 3. Analytics
- Identify drop-offs
- Find bottlenecks
- Track errors
- Monitor usage

### 4. Support Tickets
- Analyze common issues
- Identify patterns
- Track complaints
- Review feedback

## Pain Point Prioritization

### Impact vs Effort Matrix

| Pain Point | Impact | Effort | Priority |
|-----------|--------|--------|----------|
| Issue 1 | High | Low | High |
| Issue 2 | High | High | Medium |
| Issue 3 | Low | Low | Low |
| Issue 4 | Medium | Low | High |

### Prioritization Criteria
- **Frequency**: How often does it occur?
- **Severity**: How severe is the impact?
- **User Impact**: How many users are affected?
- **Business Impact**: What's the business impact?
- **Effort**: How hard is it to fix?
```

## Quick Start

### User Interview Template

```markdown
# User Interview: [Feature/Product]

## Objectives
- Understand user needs for [feature]
- Identify pain points
- Gather feedback on [aspect]

## Questions

### Opening
1. Tell me about yourself and your role
2. How do you currently [do related task]?

### Main Questions
3. What's the biggest challenge with [current solution]?
4. How would you ideally [accomplish goal]?
5. What would make this easier for you?

### Closing
6. Is there anything else you'd like to share?
7. Can we follow up if we have more questions?
```

## Anti-patterns

### ❌ Don't: Leading Questions

```markdown
# ❌ Bad - Leading question
"Do you think this feature is amazing?"
```

```markdown
# ✅ Good - Neutral question
"What do you think about this feature?"
```

### ❌ Don't: Small Sample Size

```javascript
// ❌ Bad - Too few participants
const participants = [user1, user2]  // Only 2 users!
```

```javascript
// ✅ Good - Adequate sample
// Qualitative: 5-8 participants per user segment
// Quantitative: 30+ for statistical significance
```

### ❌ Don't: Ignore Negative Feedback

```markdown
# ❌ Bad - Only positive feedback
## Research Findings
- Users love the design ✅
[Ignores all negative feedback]
```

```markdown
# ✅ Good - All feedback considered
## Research Findings
- Users love the design ✅
- Navigation is confusing ⚠️ (needs improvement)
- Feature X is hard to find ❌ (critical issue)
```
